## 1. 背景（Context）

针对"记忆与知识检索"做了三轮深度审计 + 一份外部 agent 的对标报告综合评审，核实出以下**硬事实**：

1. **权威/医学库是空的**：线上 [authoritative_kb.json](../../src/hermes_cgm_agent/knowledge/authoritative_kb.json) 只有 3 条**手写摘要**（220–278 字符/条）。一次"3→38 条"的扩充（commit `493e484`）同样是**凭模型记忆写的摘要**，2 分钟后即被回滚（`9ef7ceb`）。手写摘要 = 让 LLM 复述指南 = 幻觉源，对"零容错医学依据"不可接受。
2. **10 篇真指南是孤儿**：278 页、~103 万字符的 PDF 已抽取成 [_extracted/](../../src/hermes_cgm_agent/knowledge/_extracted/) 纯文本，但全仓**零引用**，未接入检索路径。
3. **PDF 高度多模态**（量化实测）：ESC 98 页 / 191 图 / 5116 矢量绘图 / 85 表标题；ISPAD full-3 41 页 / **11263 矢量绘图**；ADA-full 45 页只抽出 **8KB 文本**（内容几乎全在图里）。Battelino 第 16 页的 TIR→A1C 对照表被纯文本抽取后**列结构彻底丢失、数字串行成乱码**。结论：**naive 文本抽取丢掉/打乱了最高价值、最高风险的内容**（阈值表、决策流程图）。
4. **检索过度工程化**：默认 dense 嵌入是 `HashingEmbedder`（SHA1 词袋伪向量，非语义，且真语义嵌入默认关闭）；对画像（几行）、假设（个位数）这类结构化小库也跑 BM25+RRF+reranker，杀鸡用牛刀。
5. **幽灵文档债**：`MEM-ARCH-20260601`、`DECISION_LOG` 被代码引用 10+ 处，但磁盘上**不存在**（基线提交 `0cbb97f` 自述"recorded out-of-tree in dveps/docs"，从未入库）。代码指向的"单一事实源"是鬼。

并确立了一个**根本性需求**：记忆必须分为 **医学记忆** 与 **个人记忆**，二者是**反向生命周期**——
- **医学记忆**：少、静态、**不可变、零容错**、作为系统的权威医学依据；
- **个人记忆**：随时间**无界增长**、会演变、需巩固/衰减/遗忘。

## 2. 决策（Decision）

### 2.1 两类记忆按**策略**隔离（不只是打标签）

| 维度 | 医学记忆 | 个人记忆 |
|---|---|---|
| 体量 | 小、有界 | 无界增长 |
| 可变性 | 不可变、版本化 | 可变、巩固/衰减/遗忘 |
| 写入者 | **离线策展 + 临床核验；Agent 永不写入** | Agent 自动写（抽取/巩固） |
| 容错 | 零容错 | 容错（带置信度的信念） |
| 冲突 | 新指南双时间取代旧指南（两版皆留 + 生效日期） | 新证据更新置信度 / supersede |
| 检索 | 精确、逐字 + 页码引用 | 模糊、重召回、置信度加权 |
| 相撞 | **医学永远胜**，但温和呈现 | 让位于医学 |
| 信任哲学 | "已核验的真理，可被引用为权威" | "提示而非真理，用前验证" |

### 2.2 三层记忆：Hot / Warm / Cold（取代"4 层统一检索"）

- **Hot（上下文窗口）**：近 7 天血糖+事件、用户画像（全量几行）、活跃假设（全量个位数）。**SQLite 直取 → 拼文本注入 prompt，无检索层。**
- **Warm（合成状态 / "做梦"）**：后台日/周从原始数据合成结构化摘要（"近 3 天晚餐后偏高""本月 TIR 72%，环比 +3%"），prefetch 时注入。这是现有 consolidation 的**升级正名**，非新增。
- **Cold（原始档案）**：全部血糖历史、对话历史、医学卡库全文。**按需检索，不主动注入。**

### 2.3 医学库：结构化解析 + 核验论断卡（**核心决策，否决外部报告的 FTS5-裸分片方案**）

- **离线管线**：版面感知解析（Docling / VLM）把**表格输出为结构化**（HTML/Markdown，保住行列）、图与图注绑定、**流程图由 VLM 转写为结构化决策规则**。
- **知识单元 = 逐字论断卡（claim card）**：每张卡 = 一条主张（阈值/目标/规则）+ `{来源, 页码/章节, 适用人群, 原始表格/图的忠实结构, kb_version}`。**安全关键卡（阈值、低血糖处置、滴定算法）必须人工/临床核验。**
- **运行时**：检索返回逐字原文 + 出处；生成层被约束为"引用并标注"，**不许凭记忆改写**。
- **检索机制**：FTS5 索引 over 论断卡是可接受的机制细节；**但关键不在 vector-vs-FTS5，而在"卡片来自结构保真解析 + 临床核验"。**

#### 2.3.1 非专家建库工作流（D041）

本项目的操作者不是医学专业人员，因此默认工作流不要求人工撰写临床 claim。操作者只运行离线流水线：

`PDF → 分页文本/页面 PNG → Hermes CLI 结构化抽取 → 机器质量过滤 → review queue → verified=false 合入生产 KB → kb-validate/eval-rag`

Hermes 文本抽取使用 `hermes chat -q ... -Q --max-turns 1`；表格、图和低文本量页面按页渲染为 PNG，并用 `--image` 交给 Hermes 挂载的多模态模型读取。所有自动合入的卡必须保持 `verified:false`，并在运行时标注为待核验指南摘录；`verified:true` 仍需要外部临床或人工签核。

### 2.4 跨语言语义桥（保留，**否决"FTS5 单路、全删语义"**）

中文用户 ↔ 英文指南（"低血糖"≠"hypoglycemia"），纯 FTS5/BM25 词法匹配会漏召回。保留方案：**论断卡存中英双语** + 保留可选语义嵌入（`paraphrase-multilingual-MiniLM`）做跨语言召回。

### 2.5 安全闸（贯穿）

- **单向写保护阀（代码层硬 guard，非约定）**：个人记忆**永不能**写入医学记忆；医学事实可被个人推理引用，不可被合并/改写。
- **冲突仲裁**：个人信念与医学事实冲突 → 医学胜，生成层温和呈现（不否定用户）。

### 2.6 个人记忆升级（保留现有 L1/L2/L3，升级三点）

① supersede 升级为 **双时间有效期**（valid_from/valid_to）；② 强制 **溯源 lineage**（L2/L3 必回指支撑的 L1 episode）；③ 增长控制靠巩固+衰减+遗忘（已有），不走 mem0 式无界累积。

## 3. 备选方案（Alternatives Considered）

| 方案 | 来源 | 裁决 | 理由 |
|---|---|---|---|
| A. 维持 4 层统一向量检索 | 当前实现 | ❌ | 对结构化小库过度工程化；dense 退化为哈希词袋 |
| B. PDF→段落分片→FTS5、砍掉所有语义/RAG、~800 行 | 外部 agent 报告 | ⚠️ 部分采纳 | **简化方向对**（Hot 直取、删 hash/RRF、做梦合成）；**但"裸分片喂 FTS5"重蹈裸抽取覆辙——FTS5 检索乱码表格 = 向量检索乱码表格，对医学零容错有幻觉风险；且"FTS5 单路"破坏跨语言召回。作者未读 PDF。** |
| C. 结构化解析+核验论断卡 + Hot/Warm/Cold + 跨语言桥 + 单向写保护 | 本 ADR | ✅ 采纳 | 把工程预算从"运行时多模态向量管线"挪到"离线一次性解析+策展"；对 ~10–50 篇静态高危语料性价比最高、安全性最高 |

外部报告的正确部分已并入方案 C（见 §2.2 删冗余、§2.2 Warm 做梦）。方案 C ≈ "采纳报告对的 70%，否决踩在医学零容错上的 30%"。

## 4. 后果（Consequences）

**保留**：SQLite 存储层、domain 模型、analytics、EvidenceRef 溯源、safety router、dual-track 隔离、现有 consolidation 的巩固/衰减/遗忘、跨语言语义能力。

**删除/简化**：HashingEmbedder；小库上的 RRF/CrossEncoder；对 Hot 数据的检索路径（改 SQL 直取）；4 层统一检索 → 3 层。

**新增**：版面感知解析管线（离线）；论断卡 schema + 临床核验流程 + kb 版本化；Warm "做梦"后台任务；单向写保护 guard；个人记忆双时间 + lineage；**本批文档（ADR / DECISION_LOG / MEM-ARCH）以偿还幽灵文档债**。

**风险**：① 论断卡核验需要领域/临床判断，是人力瓶颈（缓解：VLM 出草稿、人核高危卡、长尾自动兜底）；② 删检索层若误删跨语言能力会伤召回（缓解：§2.4 明确保留）；③ 文档与代码须同步推进，否则继续漂移（缓解：见 §5 规矩）。

## 5. 修正计划（Correction Plan）与文档规矩

- **P1 文档先行（本 ADR + DECISION_LOG + MEM-ARCH 落盘）** ← 当前阶段
- **P2 低风险简化**：删 hash/小库 RRF；4→3 层；Hot SQL 直取。
- **P3 医学库重建**：结构化解析 → 论断卡 + 临床核验 → 版本化 + 页码引用。
- **P4 Warm "做梦"**：consolidation 升级为后台日/周合成 + prefetch 注入。
- **P5 个人记忆升级**：双时间有效期 + 溯源 lineage。
- **贯穿**：单向写保护 + 冲突医学胜。

**文档规矩（硬性）**：代码中引用的任何设计文档/决策编号，必须在仓内存在对应文件；每个重大架构决策一份 ADR；`DECISION_LOG.md` 持续追加；本 ADR 一经实现变更即更新"状态"。

## 6. 必须扭转清单（Must-Reverse）

1. 不可逆停掉"手写摘要"做医学库。
2. 停掉对所有层跑统一向量检索；Hot 改 SQL 直取。
3. 停掉把医学库当普通可搜索 Cold；按版本化/只读/可引用/零容错/单向写保护对待。
4. 删 HashingEmbedder + 小库 RRF/reranker。
5. consolidation 正名补全为 Warm 合成。
6. 补齐幽灵文档（本批）。

**不可跟随报告一起砍掉**：跨语言语义桥；dual-track 隔离 + EvidenceRef 溯源；现有巩固/衰减/遗忘。
