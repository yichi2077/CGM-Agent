## 一句话摘要

主链路（数据→分析→报告→记忆）已跑通、183 测试全绿；但我们在深审中发现**作为医学 Agent 最关键的"权威知识可信度"是空心的**，因此本阶段从"继续加功能"转向"夯实可信内核"——这会让短期对外新功能放缓，但显著提升产品的核心价值（医学建议可信、可追溯、零幻觉）。

## 一、我们在做什么

一个以 Hermes 为外壳、面向 CGM（连续血糖监测）用户的 AI 陪伴 Agent：导入血糖与生活数据 → 分析 → 生成面向「自己 / 医生 / 家人」的报告，并具备长期记忆与权威医学知识支撑。

## 二、当前进度（事实陈述）

- 数据 → 分析 → 报告 → 记忆主链路打通，G0–G8 全部落地，**183 个自动化测试全绿**。
- 安全路由（血糖红 / 黄 / 绿三区拦截）已上线，并已修正一处单位换算缺陷（避免 mmol/L 数据误判）。
- Dexcom 云端（API v3）接入管线完成，含离线 mock 全链路验证；中文受众化报告、知情陪伴者人格已实现。

## 三、发现的核心问题（为什么必须转向）

1. **医学可信度脊柱是空的**。系统宣称"双轨权威 RAG"，但线上权威库实际只有 **3 条人手写的摘要**（凭模型记忆复述指南）——对"零容错医学依据"而言这是**幻觉风险源**。10 篇真实临床指南（ADA / ISPAD / ESC / Battelino 等）虽已下载，但**从未真正接入**检索。
2. **指南是图表 / 流程图密集的多模态文档**。此前的纯文本抽取**丢失或打乱了最高价值内容**——例如关键的 TIR 目标表被抽取后数字串行成乱码、某指南 45 页只抽出约 8KB 文字（其余在图里）。结论：**"换个检索引擎"解决不了，问题在解析与结构保真**。
3. **记忆检索过度工程化且用错位置**。对只有几行的用户画像、个位数的假设也跑重型向量检索；而真正需要"会演变、会遗忘"的个人记忆反而缺少时间维度。
4. **文档与代码漂移**。代码引用的设计文档此前从未入库（幽灵文档），团队无法对齐。

## 四、处理方案（已决策）

- **记忆分两类、反向治理**：
  - **医学记忆**（少、不可变、零容错、可逐字引用）→ 走"离线结构化解析 + 临床核验 + 版本化论断卡"。
  - **个人记忆**（无界增长、会演变）→ 保留并升级现有分层巩固（加时间有效期与溯源）。
  - 两者**物理隔离 + 单向写保护**（个人永不能改写医学），冲突时医学优先但温和呈现（不否定用户）。
- **知识接入方式转变**：停止"让 AI 凭记忆写摘要"，改为对真实 PDF 做版面感知解析（表格结构化、流程图转决策规则）→ 人工核验高风险条目 → 带页码引用的中英双语卡片。
- **检索做减法**：高频结构化数据直取注入，长文档才检索；增加后台"做梦"式状态合成（业界数据：事实召回可从约 41% 提升到约 83%）。
- **文档先行**：架构决策、决策日志、记忆架构规范已全部入库，幽灵文档债已偿还。

## 五、对产品 / 排期的影响

| 维度 | 影响 |
|---|---|
| 范围 | 从"再加功能"转为"夯实可信内核"；**短期对外新功能放缓** |
| 核心价值 | 医学建议的**可信度与可追溯性显著提升**（每条结论可溯源到指南页码） |
| 新增关键依赖 | 医学论断卡需要**临床 / 领域人力核验**安全关键条目（阈值、低血糖处置、滴定算法）——新的人力瓶颈 |
| 不受影响 | 数据接入、分析、报告、安全拦截、Dexcom 同步等已稳定模块继续可用 |

**主要风险与缓解**：
- ① 解析 + 核验人力密集 → 缓解："AI 出草稿 + 人核高危卡 + 长尾自动兜底"分摊。
- ② 个人记忆历史数据的 schema 迁移需谨慎 → 缓解：迁移脚本 + 回归测试。
