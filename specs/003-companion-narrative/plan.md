# Implementation Plan: Companion Narrative + Negotiated Interaction (F4)

**Branch**: `003-companion-narrative` | **Date**: 2026-06-09 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/003-companion-narrative/spec.md`

## Summary

Bring the CGM agent's report narrative and hypothesis interaction to SOUL.md
quality by:

1. **C1 — Narrative layer enrichment** (`builder.py`): Upgrade all report sections
   (daily card, overview, metrics, observations, follow-up, patterns, doctor
   appendix) with persona-compliant conversational Chinese for SELF audience,
   simplest possible language for FAMILY, and keep clinical structure for
   CLINICIAN. Translate TIR/TAR/TBR/MBG/CV/GMI into life-language equivalents.
   Enforce SOUL.md output length norms.

2. **C2 — Hypothesis narrative templates** (`builder.py`): Wire the existing
   HypothesisState machine (CANDIDATE → OBSERVING → STABLE → ARCHIVED) into
   persona-compliant conversational templates within the `_patterns_section`
   and a new `_hypothesis_narrative` method family. No causal/assertive language.

3. **C3 — Escalation concern strategy** (`scheduler.py` + `builder.py`): Add
   consecutive-anomaly-day tracking in the scheduler's `push_tick`, expose an
   escalation level (NORMAL/CONCERN/EXTERNAL_SUPPORT) that the report builder
   reads to render progressive concern language. Vulnerable populations from
   L2ProfileItem receive earlier escalation.

## KNOWN GAP: `vulnerable_population` upstream dependency (analyze A1)

**Risk**: FR-008 / US3-AS3 read a `vulnerable_population` key from `L2ProfileItem`, but **no upstream process currently writes it** (verified: zero occurrences in the codebase). The earlier-escalation path for vulnerable users is therefore **dormant in production** and exercised only via injected test fixtures.

**Impact**: Standard-timeline escalation works for everyone; vulnerable users do not get earlier escalation until the key is populated. The fallback (missing flag → standard timeline) is safe (no error).

**Mitigation**: F4 ships the read path + tests. Populating the key (clinical/profile onboarding) is tracked separately. `_read_vulnerable_flag` defaults to `False` on a missing key.

**Owner**: Profile/onboarding work (out of F4 scope).

## On-demand escalation data-flow (analyze D1)

`builder.generate()` takes only a `ReportInput`; the push-path escalation level would never reach `reports.generate`. Resolution: optional `consecutive_anomaly_days`/`escalation_level` fields on `ReportInput`, populated by BOTH the push path and the `reports.generate` executor (via `PushSchedulerService.consecutive_anomaly_days`). See tasks T042b.
