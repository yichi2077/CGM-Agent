# Implementation Plan: Hermes Runtime Usability (F1)

**Branch**: `001-hermes-runtime-usability` | **Date**: 2026-06-08 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/001-hermes-runtime-usability/spec.md`

## Summary

Make the CGM agent usable through Hermes by closing three root-cause gaps that
are confirmed in the current code:

1. **Split-brain database** — `AppConfig.from_env()` (config.py:92) hardcodes the
   project-local `.runtime/app.db` and bypasses `resolve_database_path()`, while
   the `cgm`/`cgm_memory` plugins already resolve `~/.hermes/cgm-agent/app.db`. CLI
   writes one store; the agent reads another. Fix: route `from_env()` through
   `resolve_database_path()`, derive the storage key from the DB directory, add a
   user-initiated migration command + legacy-store detection prompt.
2. **Dangling tool schemas** — `_object_schema()` never emits a `$defs` block, so
   `events.create` (input `$ref:#/$defs/UserEvent`) and `timeseries`/`aggregate`
   (output refs) are unresolved; and `_create_event` requires the model to supply
   `event_id/created_by/user_confirmed`. Fix: inline a flat `event` input schema
   (only `event_type` + `ts_start` required) and force the technical/provenance
   fields server-side before validation.
3. **Memory tools reachability** — `cgm/__init__.py:18` excludes
   `memory.confirm/correct`; they are exposed only through the `cgm_memory`
   provider's `get_tool_schemas()`. Fix: diagnose whether Hermes surfaces provider
   tools to the model; if not, register them on the `cgm` plugin AND suppress the
   provider's duplicate, so exactly one invocable copy exists.

Plus first-run UX: empty-store prompt + opt-in `--seed-demo` (no auto-seed).

## Technical Context

**Language/Version**: Python 3.11
**Primary Dependencies**: Pydantic v2 (domain models), `cryptography` Fernet (PHI encryption); stdlib `sqlite3`, `argparse`, `shutil`
**Storage**: local SQLite at a single canonical path (`~/.hermes/cgm-agent/app.db`), Fernet-encrypted PHI columns; key co-located with the DB
**Testing**: `unittest` (`python -m unittest discover -s tests`) on the Hermes runtime venv; CI `tests.yml` + `kb-quality.yml`
**Target Platform**: local macOS/Linux behind Hermes Agent v0.15.1 shell
**Project Type**: single project — CGM capability layer (CLI + two Hermes plugins)
**Performance Goals**: not a perf feature; preserve the existing module-level executor cache (no per-call store rebuild)
**Constraints**: offline-capable core (no network needed for path/migration/event/memory flows); DB+key `0600`; no secrets/tokens in audit payloads or logs; do not modify the `~/.hermes/hermes-agent` install tree
**Scale/Scope**: single local user; ~14–90 days of CGM points; ~15 tools
